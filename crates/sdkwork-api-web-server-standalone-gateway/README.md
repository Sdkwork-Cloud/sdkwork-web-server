# sdkwork-api-web-server-standalone-gateway

Domain: platform
Capability: webserver
Package type: Rust standalone gateway
Status: active

## Public API

- `build_router`: existing management app-api/backend-api composition.
- `run_database_migrate_only`: database migration operation.
- `run_data_plane_until`: database-independent HTTP/HTTPS application data plane with explicit shutdown.
- `run_data_plane_from_config_until`: database-independent data plane with opt-in same-topology configuration Watch and failed-candidate retention.
- `run_data_plane_with_operations_until` and `run_data_plane_from_config_with_operations_until`: equivalent data-plane entrypoints with an explicit loopback host operations listener.
- `DataPlaneOperationsConfig`: validated loopback bind and canonical metric dimensions; `from_env` returns `None` when operations exposure is not configured.
- Binary operations: `serve-management`, `db-migrate`, `validate`, and `data-plane`.

## Required SDK Surface

This runtime does not consume generated HTTP SDKs. It mounts repository-owned management route crates and executes application Web Server traffic from the compiled config port.

## Configuration

Management mode uses SDKWork typed server/database environment configuration. `validate` and `data-plane` accept an explicit config argument or `SDKWORK_WEB_SERVER_CONFIG_FILE`.

`data-plane` honors `deployment.reload`: `disabled` keeps the startup generation; `watch` publishes verified same-topology generations through a lock-free request-path pointer. Listener, TLS, connection-admission, timeout, drain, or Watch-policy changes require restart.

`SDKWORK_WEB_DATA_PLANE_OPERATIONS_BIND` optionally enables a separate loopback-only listener with `/healthz`, `/livez`, `/readyz`, and `/metrics`. It is disabled by default and rejects port zero, wildcard, private-LAN, and public binds. The listener is host policy, never an application virtual-host route. It uses canonical `SDKWORK_WEB_ENVIRONMENT`, `SDKWORK_WEB_DEPLOYMENT_PROFILE`, and `SDKWORK_WEB_RUNTIME_TARGET` dimensions and fails startup on unknown aliases.

## Deployment Profile And Runtime Target Behavior

The current executable is the standalone server gateway. The data-plane-only operation does not initialize PostgreSQL or SQLite. Cloud node-scoped snapshot consumption remains a separate requirement.

## Security

- Rustls provides TLS 1.2/1.3 and configured ALPN.
- Management app-api, backend-api, and `/metrics` share one bounded `sdkwork-web-framework` registry. Unknown routes collapse to `unmatched`; request/stage series and label bytes have hard ceilings, overflow increments dropped-series counters, and startup rejects noncanonical observability dimensions.
- The request data plane owns a separate fixed-cardinality registry. Saturating atomics record connection/request admission and active lifetimes, response classes, upstream attempts/results/rejections, reload outcomes, WebSocket tunnel lifecycle, streaming Body/tunnel bytes, fixed protocol failures, and fixed DNS results. Fixed seconds histograms retain request ownership through response Body completion and upstream ownership through response Headers. A scrape aggregates current immutable target health, upstream request/physical-connection capacity, explicitly configured target physical capacity, and resource-pressure reason bits without retaining history or exposing listener/upstream ids, domains, paths, addresses, revisions, tenants, users, requests, or traces.
- Configuration and TLS PEM readers are bounded before JSON or Rustls parsing.
- HTTPS startup builds an immutable Exact/Wildcard SNI index only after every leaf certificate is currently valid, its SAN covers the declared names, and its private key matches. Unknown or absent DNS SNI fails closed.
- Reload logs revision identifiers and classified failures only; it never logs candidate bytes or private-key material.
- The accept loop acquires process and listener connection permits with non-queuing `try_acquire_owned` before creating a task, TLS state, or HTTP protocol state. Rejected sockets create no task; the listener-owned `JoinSet` therefore has a hard active-task bound, and its biased loop reaps completed tasks before accepting more sockets.
- One process-wide non-queuing request Semaphore is shared across listeners. `maxConcurrentRequests` saturation returns bounded `503`/`Retry-After`; HTTP/1 requests connection close, while HTTP/2 rejects only the excess Stream.
- With `deployment.resourcePressure`, the request gate uses one total Semaphore plus a smaller business Semaphore. Bounded URI/route classification allows only `GET`/`HEAD` exact `/healthz`, `/readyz`, or `/livez` fixed responses to bypass business capacity; every other action remains pressure-governed. Pressure/capacity rejection releases total capacity before returning, and successfully admitted work never exceeds `maxConcurrentRequests`.
- An admitted permit is held by a zero-copy response Body wrapper until streaming completion, error, H2 reset, or Body drop. Handler completion alone does not release proxy/static response capacity.
- The same Body wrapper enforces `responseBodyIdleTimeoutMs` without collecting Frames. Only non-empty Data or Trailer Frames reset progress; empty Data cannot keep a permit alive.
- One request Body wrapper enforces `requestBodyStartTimeoutMs` and `requestBodyIdleTimeoutMs` for fixed, redirect, static, and proxy actions without collecting or copying Frames. Empty Data cannot keep a request alive; expiry returns `408`, closes HTTP/1, and isolates HTTP/2 to the affected Stream.
- A post-TLS accepted-stream wrapper enforces `connectionWriteTimeoutMs` independently for continuously Pending write, flush, and shutdown operations. Reads and successful writes pass through unchanged.
- A protocol-aware Stream/Service pair enforces `http1KeepAliveIdleTimeoutMs` only between HTTP/1 requests. Response Body leases and pending-flush state prevent active uploads, streaming responses, and Pipeline work from being mistaken for idle; negotiated H2 bypasses the policy.
- The original-wire Guard and a paired Service share one connection-local atomic count for `http1MaxPipelineDepth`. Complete request heads consume a slot before Hyper sees them, synchronous dispatch releases it, excess closes the connection, and no request Body or response is queued by the guard. TLS H1 is covered after decryption; H2 bypasses the state.
- HTTP/1 parser bytes, request-line/method/request-target bytes, individual Header/Trailer name/value bytes, header count, and header-read time are explicit; HTTP/2 stream, decoded/encoded header, reset, flow-control, send-buffer, Frame-rate, new-Stream-rate, and Continuation values are bounded before serving.
- One allocation-free request URI scanner applies raw/decoded Path, segment, Query string, parameter, and name/value component budgets across H1/H2 before route selection. It rejects malformed percent escapes and decoded NUL/control/backslash; valid proxy Query bytes remain unchanged.
- A bounded original-wire Framing Guard runs after TLS and before Hyper, supports exactly Chunked input, validates Trailers, and rejects TE/CL plus duplicate Content-Length before normalization. Proxy requests are reframed by the Hyper upstream client after client framing headers are removed.
- A constant-memory HTTP/2 Wire Guard runs after TLS ALPN and before Hyper, validates Preface/Frame metadata, counts fixed-window Frame/new-Stream/`RST_STREAM` churn, and bounds encoded Header Blocks without retaining payloads or Header values. Wire violations close only the offending connection; H2 retains protocol and GOAWAY ownership.
- Hyper/H2 sends an idle PING at `http2KeepAliveIntervalMs`; missing ACK at `http2KeepAliveTimeoutMs` produces `GOAWAY(NO_ERROR)` and connection close. A compliant idle peer remains connected, HTTP/1 bypasses the policy, and SDKWork adds no parallel PING state machine or task.
- Every accepted connection has one `maxConnectionAgeMs` deadline. Expiry asks Hyper to stop HTTP/1 reuse or send H2 `GOAWAY(NO_ERROR)`, polls accepted work for at most `drainTimeoutMs`, then cancels the supervised connection task. Listener shutdown uses the same owned task registry; no accepted connection task is detached.
- Hyper advertises configured maximum concurrent Streams, Frame size, and decoded Header List size. HPACK dynamic-table sizing is not exposed by the selected server builder, so no unsupported application field exists and H2 0.4.15 retains its finite 4,096-byte default.
- Fixed, Chunked, proxy, non-proxy, and HTTP/2 bodies without Content-Length are stream-counted against the application Body limit; no Body-sized collection is introduced.
- Reverse-proxy request and response bodies preserve Data and Trailer frames. Trailer declarations and actual frames share finite count/byte limits and forbidden-field checks; only canonical `TE: trailers` is regenerated toward upstreams.
- HTTP/1 enables write-side half-close, accepts only HTTP/1.1 `Expect: 100-continue`, returns early `413` without `100` for known fixed-length overflow, rejects unsupported expectations with `417`, and removes `Expect` before proxying upstream. HTTP/1.0 default-host, Keep-Alive, ordered Pipeline, finite Pipeline read-ahead, and Transfer-Encoding rejection are raw-socket tested.
- Static traversal/symlink escape checks run before `tower-http` file service.
- Proxy bodies are streamed with a counted hard limit; redirects and protocol upgrades are not followed silently.
- Classic HTTP/1.1 WebSocket proxying requires GET, exact Connection/Upgrade tokens, and no Body framing. Hop-by-hop fields are sanitized and canonical upgrade fields regenerated; upstream non-101 responses remain bounded streaming proxy responses, while malformed upstream 101 metadata becomes a generic local `502` and closes the downstream connection.
- Successful WebSocket/WSS tunnels are owned by one runtime supervisor after Hyper releases the upgraded sockets. Each task holds upstream request admission, physical connection capacity, and its immutable Watch generation; uses fixed 16 KiB directional buffers; observes `maxConnectionAgeMs` and runtime shutdown; and consumes only the remaining shared drain budget. Directional byte totals use Tokio's authoritative successful copy result; partial bytes after copy error are not reported because the API does not expose them. HTTP/2 extended CONNECT, frame parsing, heartbeat, and exact Nginx WebSocket timeout directives are not claimed.
- Upstream early responses pause incomplete uploads without draining or collecting them. Complete response handoff cancels the upstream producer; HTTP/1 explicitly closes the downstream connection, while H2 uses `RST_STREAM(NO_ERROR)` and preserves the connection for other Streams.
- Hop-by-hop headers and untrusted forwarding identity are removed/replaced. An optional listener-local `trustedProxy` policy considers exactly one bounded `X-Forwarded-For` field only when the direct peer matches an explicit CIDR. Non-recursive/rightmost and recursive/right-to-left trusted-hop selection match the Nginx realip intent; malformed, duplicate, over-byte, and over-hop trusted input returns bounded `400`. The effective address becomes one canonical upstream XFF value and never a metric label or current access-log field.
- Optional listener-local `proxyProtocol` parses mandatory HAProxy PROXY v1/v2 only after connection admission and only from an immediate peer in an explicit trusted CIDR. Parsing precedes TLS ClientHello and HTTP, is covered by a finite timeout and total Header ceiling, uses fixed stack buffers, and accepts v1 `TCP4`/`TCP6`/`UNKNOWN` and v2 `LOCAL` or `PROXY` over TCP4/TCP6. Every v2 TLV is structurally validated and streamed through fixed scratch memory; the additive `crc32cPolicy` can ignore values, validate one when present, or require one. CRC32C detects corruption but does not authenticate the peer, and no other TLV is interpreted or trusted. The resolved source becomes the same effective client identity used by canonical upstream XFF, IP-hash, and safe retries. Missing, malformed, oversized, untrusted, disabled-version, or CRC-policy-failing Headers close without an HTTP response, and policy changes are Restart-only.
- One bounded asynchronous system-resolver runtime is shared per resolver profile. Non-queuing admission, finite lookup timeout, and `maximumAnswers + 1` retention prevent resolver waiter and answer growth; an upstream-specific immutable policy rechecks every answer before the Hyper connector receives it.
- Reverse proxy destinations default to public unicast. Explicit narrow CIDRs may authorize loopback/private/shared/link-local/ULA targets, but cloud metadata and hard special-use addresses always fail closed. Mixed permitted/forbidden answer sets fail as a whole, and the configured hostname remains Host/SNI/certificate authority.
- HTTPS upstreams use verified system WebPKI trust by default. Per-upstream TLS policy can replace or extend system roots, require a protected-file client identity, and constrain TLS 1.2/1.3. All TLS inputs are read through the one-MiB file bound, custom roots are count-bounded, and no hostname-verification bypass, SNI override, or plaintext fallback exists.
- Each immutable upstream owns one complete Hyper/Rustls client and connection pool, isolating DNS/address policy, trust roots, mTLS identity, versions, and timeout state. Rustls uses an explicit AWS-LC provider even when workspace dependencies also enable Ring. Watch candidates build this complete security context before `ArcSwap` publication; failure retains the active client/pool and valid policy changes cannot reuse the previous pool.
- Each upstream also owns a non-queuing `maxInFlightRequests` Semaphore. Its permit is acquired before target/DNS/TLS work and remains attached to the streaming downstream Body through completion, error, reset, cancellation, or drop. Saturation returns fixed local `503`/`Retry-After` without polling the client Body or starting upstream work.
- Optional `upstream.retry` performs at most eight sequential attempts within one total deadline and one retained request permit. Only Body-end-of-stream idempotent methods are eligible; POST, PATCH, pending Body/Trailer Frames, and WebSocket upgrades never replay. Fixed transport/timeout/502/503/504 conditions select a different currently healthy target through a stack bitmap, while local capacity saturation and client Body failures remain terminal.
- Upstream `maxConnections` adds a non-queuing aggregate physical-connection Semaphore per immutable generation. Optional target `maxConnections` adds a second authority-owned Semaphore after validation rejects ambiguous duplicate origins. Both permits live in the TCP I/O through TLS, HTTP/1 activity, HTTP/2 multiplexing, and idle pooling. Local saturation does not mutate target health; idle expiry, Watch replacement, and shutdown close sockets and release ownership. Dual-stack connection attempts remain sequential and use a bounded non-final-address fallback inside the total connection timeout.
- The upstream Hyper client applies `maxResponseHeaderBytes` to HTTP/1 parser growth and the H2 Header List limit, and applies non-default `maxResponseHeaders` to the HTTP/1 parser. Before proxy response construction, one allocation-free protocol-independent pass counts every decoded field occurrence as name bytes, value bytes, separators, and the terminating CRLF with checked arithmetic. Count/byte overflow drops the unpolled response Body, records target failure, and returns only the bounded local `502 upstream failed` response; no rejected Header name or value is logged or forwarded.
- Passive health uses fixed per-target atomics for consecutive failures, ejection deadline, and one half-open probe. Configured `5xx` and transport failures eject only the affected target; healthy alternatives continue round-robin service. All-ejected/probing state fails locally without retries, and compare-exchange prevents stale concurrent success from clearing a newer ejection.
- Business selection first chooses the primary tier whenever any unattempted primary is active/passively eligible; otherwise it chooses the backup tier. Default `round-robin` performs a linearized smooth weighted transaction in stable configuration order, producing exact stable sequences such as `A,A,B,A` for 3:1 while retaining independent primary/backup phases. Each generation stores two boxed fixed-cardinality scalar arrays behind one short standard Mutex; the critical section does bounded scans and target-atomic operations only, never I/O or `.await`. Request-local retry exclusion skips without erasing global phase, health transitions reset target phase atomically with eligibility changes, and a half-open claim race performs at most one bounded fallback. No tier vector, expanded schedule, request collection, async waiter, or lock crosses request execution or Body streaming. Process-lifetime lock contention uses one fixed-label counter.
- `upstreams[].loadBalancing = least-connections` minimizes generation-local active business requests divided by weight inside that same eligible tier with overflow-safe integer cross multiplication. One RAII target lease spans setup, each retry attempt, downstream streaming Body or rejected-upgrade Body, and accepted WebSocket tunnel lifetime. HTTP/2 Streams are counted independently from pooled physical connections. Counters are fixed per configured target, Watch generations never share them, and no request queue, expanded schedule, target-state copy, dynamic map, or lock across I/O is introduced.
- `upstreams[].loadBalancing = random-two-least-connections` uses one generation-local atomic SplitMix64 sequence to sample two distinct eligible targets by effective weight without replacement, then applies the same weighted active-request comparison. Multiply-high bounded mapping, a fixed number of scans, two claim attempts, and at most one least-connections fallback add no request allocation, RNG lock, queue, or resampling loop. The PRNG is scheduling-only and not cryptographic; Watch does not share its state.
- `upstreams[].loadBalancing = ip-hash` uses the listener-resolved effective client IP, which is the direct accepted peer unless an explicit trusted-proxy policy succeeds. It preserves the Nginx IPv4-three-byte/IPv6-sixteen-byte key, recurrence, nominal weighted ranges, twenty-one total mappings, and one smooth fallback. Health-unavailable targets remain in the hash range but cannot be selected; retries use the same effective key and fixed attempted bitmap. The path stores no client state or ring and emits no client-IP telemetry. Config compilation rejects `slowStartMs` because Nginx disallows it with `ip_hash`.
- Optional target `slowStartMs` ramps the effective integer weight from one slot to nominal after fully eligible active or passive health recovery. The same effective weight drives round-robin and least-connections. One immutable duration plus one atomic monotonic start offset is stored per target; initial startup/Watch state is nominal, repeated failure clears the ramp, repeated recovery restarts it, and stale completion cannot clear or serve at nominal through a newer ramp.
- Optional active health runs in exactly one supervised Tokio scheduler task per immutable generation. One fixed deadline entry exists per checked target, concurrent probe futures are bounded by `maxConcurrentHealthChecks`, one target never overlaps, and response Bodies are read only through the configured byte ceiling. Probes reuse the upstream's guarded DNS/TLS/mTLS client context without consuming business admission permits.
- Active and passive state remain independent atomics: active failure thresholds remove a target, consecutive active successes restore it, active success cannot clear passive ejection, and business success cannot clear active unavailability. Watch starts the candidate scheduler before publication and cancels/joins the previous one; shutdown cancels and joins all current health work.
- Optional resource-pressure admission runs one supervised, non-overlapping sampler. Windows reports Working Set/HANDLE count; Linux uses bounded RSS/FD/host-limit reads and finite cgroup v2 memory. Absolute reserves, distinct effective thresholds, consecutive-sample hysteresis, and event-loop wake lag drive one atomic state. Pressured new sockets close before task/TLS/HTTP creation, established business requests receive bounded `503`/`Retry-After`, and exact fixed operations routes remain available on established connections. Dynamic host/cgroup capacity contraction is revalidated, and shutdown joins the sampler.
