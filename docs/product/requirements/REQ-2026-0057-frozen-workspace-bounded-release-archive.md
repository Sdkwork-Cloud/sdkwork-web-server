## Frozen Dependency Boundary

The application-root `pnpm-workspace.yaml` is the native package workspace authority used after the
reusable workflow checks out `sdkwork.workflow.json` dependencies as siblings. The root lockfile is
generated by the package manager and the release install uses `--frozen-lockfile`; CI must fail rather
than mutate dependency resolution. Workflow dependency declarations remain checkout authority and do
not replace Cargo or package-manager manifests.

The frozen install proves only that the declared source graph can be resolved reproducibly in the
assembled workspace. The root workspace explicitly allowlists the trusted `esbuild` preparation
script required by the Vite build graph; new native build scripts must be reviewed and added
explicitly rather than being silently executed. This is not an SBOM, vulnerability scan, license
review, signature, provenance, or proof that a mutable dependency ref resolved to the intended commit.

## Archive Trust Boundary

`scripts/webserver-release.mjs package` produces a deterministic profile/version-specific candidate,
atomically publishes the archive and checksum, and immediately calls the same validator used by the
explicit `validate` operation. The workflow also invokes validation as a distinct lifecycle step
before upload so a missing, replaced, or malformed artifact fails publication.

Validation does not extract the archive. It first bounds and verifies the outer archive and checksum,
then streams tar entries through the pinned `tar` parser. Every entry must match the exact package
contract and the bounded `package.manifest.json`. Entry paths, types, metadata, ownership, modes,
timestamps, declared/observed byte counts, per-file hashes, and ordering are verified before trust.

## Verification Evidence

- `pnpm install --frozen-lockfile` resolves all 24 workspace projects from the current lockfile
  without a resolution update, and the required `esbuild` build script is explicitly allowlisted.
- The paired release/profile contract run passes 11 tests with no failures; the NTFS symbolic-link
  fixture is explicitly skipped on Windows because it cannot provide portable tar link evidence.
- A real small tar fixture built through the same parser passes validation; checksum, manifest,
  duplicate/unexpected-entry, and executable-mode corruptions fail closed.
- Workspace member protocol, workflow schema/matrix, pnpm script, agent/workflow, repository docs,
  source configuration, and diff checks pass.
- `pnpm verify` passes after the parallel data-plane test port allocator was made process-unique;
  the complete 55-test data-plane integration suite also passes in four additional repeated runs.
- Windows non-dry-run packaging stops at the selected Linux architecture host gate before Cargo
  build or source repair.
- This requirement's acceptance evidence did not execute Linux archives. REQ-2026-0058 subsequently
  produced and validated the standalone/cloud Linux x64 archives and exercised packaged
  HTTP/HTTPS readiness, traffic, SIGTERM exit, and cleanup.

## Remaining Release Gates

This requirement closes dependency-resolution and archive-validation integrity only. Commercial
release still requires declared container/service targets, artifact
SBOM/signature/provenance/attestation,
vulnerability and license review, upgrade/rollback/uninstall evidence, and release approval.
