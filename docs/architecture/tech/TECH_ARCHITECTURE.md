## 1. Architecture Overview

SDKWork Web Server is a Rust HTTP control plane for multi-tenant site, domain, deployment, certificate, nginx, and edge-agent management. It exposes two SDKWork API surfaces:

- **app-api** (`/app/v3/api/*`) — tenant application clients
- **backend-api** (`/backend/v3/api/*`) — platform operators and edge agents

Runtime entrypoint: `sdkwork-web-standalone-gateway` (unified-process topology by default).

## 2. Technology Choices

| Layer | Choice |
| --- | --- |
| Language / runtime | Rust + Tokio |
| HTTP framework | Axum via `sdkwork-web-framework` |
| Database | `sdkwork-database` + SQLx (`web_*` tables) |
| ID generation | Snowflake PKs + UUID v4 via `sdkwork-database-id` |
| Shared utilities | `sdkwork-utils-rust` (envelope, crypto, datetime, slugify) |
| IAM | `sdkwork-iam-web-adapter` dual-token + agent-token modes |
| ACME | `instant-acme` (Let's Encrypt) + `rcgen` (self-signed) |

## 3. System Boundaries And Modules

```text
sdkwork-web-standalone-gateway
  ├── sdkwork-routes-webserver-app-api      (HTTP boundary)
  ├── sdkwork-routes-webserver-backend-api  (HTTP boundary + agent routes)
  ├── sdkwork-intelligence-webserver-service (business logic)
  ├── sdkwork-intelligence-webserver-repository-sqlx (persistence)
  ├── sdkwork-webserver-database-host       (DB lifecycle)
  ├── sdkwork-webserver-acme-service        (certificate issuance)
  ├── sdkwork-webserver-edge-runtime        (nginx deploy/validate)
  ├── sdkwork-web-agent                     (edge heartbeat/sync client)
  └── sdkwork-webserver-certificate-worker  (renewal scan job)
```

## 4. Directory And Package Layout

See repository `README.md` and `specs/component.spec.json` for the canonical dictionary. OpenAPI authorities live under `apis/`; materialized JSON and route manifests are generated by `pnpm api:materialize`.

## 5. API, SDK, And Data Ownership

- OpenAPI YAML is the contract authority for both surfaces.
- Success responses use `SdkWorkApiResponse`; errors use `ProblemDetail` (`application/problem+json`).
- SDK families: `sdkwork-web-app-sdk`, `sdkwork-web-backend-sdk` under `sdks/`.
- Database module id: `web`, table prefix: `web_`.

## 6. Security, Privacy, And Observability

- Production auth is fail-closed through IAM PostgreSQL resolver.
- Agent routes authenticate via `X-SDKWork-Agent-Token` (one-time bootstrap token hashed at rest).
- Secret env variables and certificate private keys stored AES-256-GCM encrypted.
- All success/error responses include `traceId` header per `API_SPEC.md`.

## 7. Deployment And Runtime Topology

- **Standalone**: single `sdkwork-web-standalone-gateway` process (`configs/topology/standalone.*.env`).
- **Cloud**: unified-process or split-services with optional platform API gateway (`specs/topology.spec.json`).
- Container image: `deployments/docker/Dockerfile`; Kubernetes manifests under `deployments/kubernetes/`.
- Packaging: `sdkwork.workflow.json` → `linux-x64-cloud-server-tar-gz`.
